# Retrieval-Augmented Generation (RAG)

## What is it?
**RAG** combines a **retrieval system** (search) with a **generative LLM**. Instead of relying solely on the model's training data, RAG fetches relevant documents at query time and injects them into the prompt.

## Why does it matter?
- LLMs have a knowledge cutoff — they don't know recent events.
- LLMs can hallucinate facts. RAG grounds answers in real documents.
- Cheaper than fine-tuning: update the document store, not the model.

## How it works

```
User Query
    │
    ▼
┌──────────────┐        ┌──────────────────────┐
│   Retriever  │──────▶│   Vector / BM25 Store │
│  (embed query│        │  (all your documents) │
│   + search)  │◀──────│                       │
└──────┬───────┘        └──────────────────────┘
       │ top-k relevant docs
       ▼
┌──────────────┐
│ Prompt Builder│  "Answer using this context: …\nQuestion: …"
└──────┬───────┘
       │
       ▼
┌──────────────┐
│     LLM      │──▶  Grounded Answer
└──────────────┘
```

## Key components

| Component | Role |
|-----------|------|
| **Chunking** | Split documents into ~256–512 token chunks |
| **Embedding model** | Encode chunks + queries into vectors |
| **Vector store** | Index vectors for fast nearest-neighbour search (FAISS, Pinecone, Chroma) |
| **Retriever** | Return top-k most similar chunks to the query |
| **Generator** | LLM that produces the final answer from the retrieved context |

## Retrieval methods
- **Dense retrieval**: embed query and docs with a neural model, use cosine similarity.
- **Sparse retrieval**: BM25 / TF-IDF keyword matching.
- **Hybrid**: combine both for best recall.

## Run the example
```bash
python rag.py
```

No external dependencies required. The example uses a simple bag-of-words embedding for demonstration.
