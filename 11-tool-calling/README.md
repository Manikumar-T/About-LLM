# Tool Calling (Function Calling)

## What is it?
**Tool calling** lets an LLM request execution of external functions — APIs, databases, calculators, code runners — during a conversation. The model decides *when* to call a tool and *what arguments* to pass; the host program executes it and returns the result.

## Why does it matter?
- LLMs are frozen at training time. Tool calling gives them access to **live data** (weather, stocks, databases).
- Models are bad at arithmetic. Offloading to a **calculator** is more reliable.
- Enables **agentic** behaviour: the model can take actions in the real world (send emails, write files, call APIs).

## How it works

```
┌─────────────────────────────────────────────────────┐
│  1. Define tools (JSON schema) and send to model    │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│  2. Model reads user message + tool schemas         │
│     → decides: answer directly OR call a tool       │
└─────────────────────┬───────────────────────────────┘
                      │ tool_call { name, arguments }
┌─────────────────────▼───────────────────────────────┐
│  3. HOST executes the tool (not the model!)         │
│     → returns result as a string/JSON               │
└─────────────────────┬───────────────────────────────┘
                      │ tool result
┌─────────────────────▼───────────────────────────────┐
│  4. Model receives result, generates final answer   │
└─────────────────────────────────────────────────────┘
```

## Tool schema (JSON)
```json
{
  "name": "get_weather",
  "description": "Get the current weather for a city.",
  "parameters": {
    "type": "object",
    "properties": {
      "city": { "type": "string" },
      "unit": { "type": "string", "enum": ["celsius", "fahrenheit"] }
    },
    "required": ["city"]
  }
}
```
The **description** is critical — the model reads it to decide when to use the tool.

## Message flow (OpenAI / Anthropic API format)

```
messages = [
  { "role": "user",      "content": "Weather in Paris?" },
  { "role": "assistant", "tool_calls": [{ "name": "get_weather", "arguments": {"city": "Paris"} }] },
  { "role": "tool",      "content": '{"temperature": "18°C", "condition": "Cloudy"}' },
  { "role": "assistant", "content": "It is 18°C and cloudy in Paris." },
]
```

## Parallel tool calls
Modern APIs (GPT-4o, Claude 3+) can request **multiple tools in one turn**:
```json
[
  { "name": "get_weather", "arguments": {"city": "London"} },
  { "name": "get_weather", "arguments": {"city": "Tokyo"} }
]
```

## Common tools in production agents

| Tool | Purpose |
|------|---------|
| Web search | Real-time information retrieval |
| Code interpreter | Run Python, analyse data |
| Database query | Fetch structured data |
| Calendar / email | Take actions in the world |
| RAG retrieval | Search a private document store |
| Calculator | Reliable arithmetic |

## Key design rules
- Write **precise descriptions** — the model uses them to route correctly.
- Return **structured JSON** from tools so the model can parse it easily.
- Validate and **sanitise all arguments** before execution (the model can hallucinate).
- Set a **max iterations** limit to prevent infinite tool loops.
