# Prompt Engineering

## What is it?
**Prompt engineering** is the practice of carefully crafting the text you send to an LLM to get better, more reliable outputs — without changing the model weights.

## Why does it matter?
The same model can give brilliant or useless answers depending on how the prompt is written. Good prompts unlock capabilities that are already inside the model.

## Core techniques

### 1. Zero-shot
Ask directly without examples. Works for simple or well-known tasks.
```
Classify sentiment: "I loved it!"
Output:
```

### 2. Few-shot
Provide 2–5 input/output examples before your real question. The model learns the format from context.
```
Input: "Great product!" → Output: Positive
Input: "Terrible!"      → Output: Negative
Input: "It was okay."   → Output:
```

### 3. Chain-of-Thought (CoT)
Ask the model to reason step-by-step before giving the final answer. Dramatically improves accuracy on math and logic.
```
Let's think step by step:
```

### 4. System / User split
Chat models (GPT-4, Claude) support separate system and user messages. Use the system message to set persona, constraints, and format.

### 5. Role prompting
Tell the model *who* it is — it will adopt that persona's vocabulary and reasoning style.
```
You are a senior security engineer. Review this code for vulnerabilities.
```

## Tips
- Be specific about the desired **output format** (JSON, bullet list, one sentence).
- Provide **constraints** (max length, language, tone).
- Use **delimiters** (triple backticks, XML tags) to separate sections of the prompt.
- **Iterate**: treat prompts like code — test, measure, refine.
